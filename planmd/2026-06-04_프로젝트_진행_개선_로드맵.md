# Vue MDN 프로젝트 진행상황 및 개선 로드맵

작성일: 2026-06-04

## 1. 문서 목적

이 문서는 현재 `vue-mdn` 프로젝트의 실제 코드와 기존 개발 문서를 기준으로, 현재까지 진행된 부분과 개선이 필요한 부분, 앞으로 진행해야 할 작업을 계획적으로 정리하기 위해 작성한다.

프로젝트는 메가스터디 관련 디자인/컴포넌트/프로젝트 산출물을 Vue 화면에서 분류별로 확인하고, 게시물 형태로 등록 및 관리하는 내부 아카이브 성격의 애플리케이션이다.

## 2. 현재 프로젝트 구조

### 2.1 전체 구조

- `frontend`: Vue 3 + Vite 기반 화면 애플리케이션
- `backend`: NestJS 기반 게시물 API 서버
- `devStatus.md`: 현재 개발 현황 정리 문서
- `devStep.md`: 향후 개발 단계 문서
- `vueConcept.md`: 프로젝트를 기준으로 Vue 개념을 학습하기 위한 문서
- `planmd`: 진행 계획 및 점검 문서 보관 폴더

### 2.2 프론트엔드 기술 구성

- Vue 3
- Vue Router
- Vite
- Options API 중심의 Vue 컴포넌트 작성 방식
- CSS 파일 분리 방식
- `localStorage` 기반 임시 게시물 저장

### 2.3 백엔드 기술 구성

- NestJS
- TypeScript
- Jest 테스트 환경
- 현재는 메모리 배열 기반 게시물 저장
- `POST /posts`, `GET /posts`, `DELETE /posts/:id` 수준의 기본 API만 구현

## 3. 현재 진행상황

### 3.1 공통 레이아웃

현재 공통 레이아웃은 기본 골격이 구성되어 있다.

- `App.vue`에서 상단 헤더, 좌측 내비게이션, 본문 `router-view` 구조 사용
- `TopHeader.vue`에서 소개, 스타일가이드, 컴포넌트, 프로젝트 메뉴 제공
- `LeftNav.vue`에서 `LeftMenu.js` 데이터를 기반으로 좌측 메뉴 제공
- 잘못된 경로 접근을 위한 `NotFound.vue` 라우트 존재

진행 판단:

- 1차 화면 IA는 구성 완료 상태
- 메뉴 경로와 라우터 경로도 기존 문서 기준으로 일부 정리된 상태
- 다만 메뉴 데이터와 화면 문자열에 인코딩 깨짐이 다수 남아 있어 실제 사용 품질은 아직 낮음

### 3.2 소개 및 스타일가이드 영역

현재 구현된 화면:

- `/intro/summary`
- `/intro/designrule`
- `/style-guide/font`
- `/style-guide/logo`, `/style-guide/color`, `/style-guide/layout`, `/style-guide/icon`은 `DefaultPage` 중심의 임시 연결

진행 판단:

- 소개 영역은 기본 화면 연결 완료
- 스타일가이드는 폰트 화면 외에는 실제 콘텐츠가 부족함
- 향후 디자인 기준 문서화와 샘플 코드 정리가 필요함

### 3.3 컴포넌트 아카이브 영역

현재 라우터에 연결된 컴포넌트 카테고리:

- 배너
- 퀵메뉴
- 게시판
- 탭메뉴
- 리뷰
- 이벤트
- 기획/선생님 탭
- 동영상
- 팝업
- 쿠폰
- 상품 리스트

현재 주요 구현:

- 각 카테고리별 목록 페이지가 존재
- `PageComponentListItem.vue`를 통해 게시물 카드/링크/CSS 코드 영역 표시
- `PostEditor.vue`를 통해 컴포넌트 게시물 등록 및 수정 가능
- `postManager.getPostsByCategory()`로 `localStorage`의 게시물을 카테고리별 조회
- 등록 후 `/component/{category}`로 이동

진행 판단:

- 컴포넌트 게시물 등록, 목록 조회, 수정, 삭제 흐름은 브라우저 로컬 기준으로 작동 가능한 형태까지 진행됨
- `Banner.vue`는 비교적 정리된 구조를 사용하고 있어 다른 컴포넌트 페이지의 기준으로 삼을 수 있음
- 일부 페이지는 변수명/문자열이 깨진 상태라 빌드 및 유지보수 위험이 큼

### 3.4 프로젝트 아카이브 영역

현재 라우터에 연결된 프로젝트 카테고리:

- 메가선생님
- 메가패스
- 수능/내신
- 대학별고사
- 입시정보
- 이벤트 프로모션
- 온라인서점
- 메가캠페인
- 일부 임시 경로: `emarket`, `project/post/:id`

현재 주요 구현:

- `ProjectListPage.vue` 공통 목록 컴포넌트 존재
- `ProjectCard.vue`를 통해 썸네일, 태그, 날짜, 게시판/PC/MO 링크 표시
- `ProjectPostEditor.vue`를 통해 프로젝트 게시물 등록 및 수정 가능
- 프로젝트 목록은 `postManager.getPostsByCategory(category, 'project')` 기준으로 조회
- 목록 페이징 기능이 일부 구현되어 있음

진행 판단:

- 프로젝트 목록은 컴포넌트 목록보다 공통화 수준이 높음
- 프로젝트 게시물 등록/수정/목록 조회 흐름은 로컬 기준으로 진행됨
- 다만 삭제/상세/관리자 관점의 통합 관리는 부족함

### 3.5 게시물 데이터 관리

현재 `frontend/src/utils/postManager.js`에서 담당하는 기능:

- `megaDbPosts` 키로 `localStorage` 읽기/쓰기
- 게시물 전체 조회
- 카테고리별 게시물 조회
- 게시물 저장
- 게시물 삭제
- `normalizePost()`를 통한 구형 필드와 신규 필드 호환
- 최신순 정렬

현재 정규화되는 주요 필드:

- `type`
- `category`
- `tags` / `labels`
- `thumbnailUrl` / `imgUrl`
- `links.board`
- `links.pc`
- `links.mo`
- `createdAt`
- `updatedAt`
- `date`

진행 판단:

- 기존 문서의 2단계인 데이터 모델 표준화는 상당 부분 반영됨
- 구형 필드와 신규 필드 호환 처리가 있어 기존 로컬 데이터가 깨질 가능성을 줄였음
- 그러나 실제 타입 정의, 검증, 서버 API 연동은 아직 부족함

### 3.6 백엔드 API

현재 백엔드 `posts` 모듈 상태:

- `POST /posts`: 게시물 생성
- `GET /posts`: 전체 게시물 조회
- `DELETE /posts/:id`: 게시물 삭제
- 저장 방식은 서버 메모리 배열
- DTO와 Entity는 빈 클래스 수준

진행 판단:

- API 골격은 존재하지만 실제 운영 가능한 데이터 관리 단계는 아님
- 서버 재시작 시 데이터가 사라짐
- 프론트엔드는 아직 백엔드 API를 사용하지 않고 `localStorage`를 사용함

## 4. 개선되어야 할 부분

### 4.1 최우선 개선: 깨진 문자열과 소스 안정화

현재 여러 파일에서 한글 문자열이 깨져 있다.

영향이 큰 파일 예시:

- `frontend/src/data/LeftMenu.js`
- `frontend/src/components/TopHeader.vue`
- `frontend/src/components/PageComponentListItem.vue`
- `frontend/src/components/ProjectListPage.vue`
- `frontend/src/pages/component/ProductList.vue`
- `frontend/src/pages/posting/PostEditor.vue`
- `frontend/src/pages/posting/ProjectPostEditor.vue`
- `devStatus.md`, `devStep.md`, `vueConcept.md`

특히 `ProductList.vue`와 `LeftMenu.js`는 깨진 문자열이 단순 화면 문구를 넘어 JavaScript 식별자나 문자열 닫힘 문제로 보이므로 빌드 실패 가능성이 있다.

개선 방향:

- 소스 파일 인코딩을 UTF-8로 통일
- 깨진 한글 문구를 정상 한국어로 복구
- JavaScript 식별자는 한글보다 영문 camelCase로 통일
- 메뉴명, 카테고리명, 라우터 path를 별도 상수 데이터로 관리
- 복구 후 `npm run build`로 즉시 검증

### 4.2 라우터와 메뉴 데이터 정합성

현재 메뉴 경로와 라우터 경로는 일부 정리되었지만, 여전히 다음 정리가 필요하다.

- `examhighscool`처럼 오타로 보이는 path 점검
- `emarket` 라우트와 `onlinebookstore` 라우트의 역할 정리
- `/project/post/:id` 임시 라우트 필요 여부 판단
- 스타일가이드의 임시 `DefaultPage` 연결 항목 처리
- 프로젝트 상세 path와 목록 path의 역할 분리

개선 방향:

- `routes`와 `LeftMenu`의 카테고리 키를 같은 기준표로 맞춤
- path 오타는 기존 데이터 영향 여부를 확인한 뒤 정정
- 미구현 페이지는 `DefaultPage`가 아니라 명확한 준비중 화면 또는 실제 콘텐츠로 전환

### 4.3 컴포넌트 목록 페이지 공통화

현재 `Banner.vue`는 구조가 비교적 명확하지만, 컴포넌트 카테고리별 페이지가 개별 파일로 반복되어 있다.

문제점:

- 삭제/수정/빈 목록 UI가 페이지마다 중복됨
- 일부 파일만 `postManager`를 정상적으로 사용하는 상태일 수 있음
- 카테고리명이 하드코딩되어 있어 오류 발생 가능성이 높음

개선 방향:

- `ComponentListPage.vue` 공통 컴포넌트 생성
- `title`, `category`를 props로 받아 목록/등록/수정/삭제 처리
- 기존 `Banner.vue`, `QuickMenu.vue`, `Board.vue` 등은 공통 컴포넌트를 감싸는 얇은 wrapper로 정리
- 빈 목록 UI, 삭제 confirm, 수정 이동 로직을 한 곳에서 관리

### 4.4 게시물 상세 보기 기능 부족

현재 게시물은 목록 카드와 수정 화면 중심으로 구성되어 있다.

부족한 점:

- 컴포넌트 게시물 상세 페이지 없음
- 프로젝트 게시물 상세 페이지 없음
- 카드 클릭 시 상세로 이동하는 명확한 흐름 부족
- CSS/HTML/JS 코드 확인 영역이 목록 내부에 치우쳐 있음

개선 방향:

- `/component/{category}/{id}` 상세 라우트 추가
- `/project/{category}/{id}` 상세 라우트 추가
- 상세 화면에서 썸네일, 태그, 담당자, 링크, 코드, 설명을 구조적으로 표시
- 목록에서는 요약만 보여주고 상세에서 전체 정보를 확인하도록 역할 분리

### 4.5 프론트엔드 CRUD 완성도

현재 로컬 저장 기반 등록/수정/삭제는 구현되어 있으나 완성도를 높여야 한다.

개선 항목:

- 저장 전 필수값 검증
- URL 형식 검증
- 이미지 미등록 시 기본 이미지 처리 기준 통일
- 삭제 전 확인 팝업 UI 통일
- 저장 성공/실패 메시지 통일
- 수정 중 취소 시 이전 페이지 또는 목록 페이지 이동 기준 통일
- `deviceType`과 `deviceTypes` 필드명 통일
- `imgUrl`과 `thumbnailUrl` 필드명 통일

권장 기준:

- 내부 저장 모델은 `thumbnailUrl`, `deviceTypes`를 표준으로 사용
- 화면 호환용 변환은 `normalizePost()` 내부에서만 처리
- 신규 코드에서는 구형 필드 직접 사용을 줄임

### 4.6 백엔드 API 고도화

현재 백엔드는 기본 구조만 있는 상태다.

필요한 API:

- `GET /posts/:id`
- `GET /posts?type=&category=&status=&keyword=`
- `PATCH /posts/:id`
- `DELETE /posts/:id`
- 추후 이미지 업로드 API

필요한 설계:

- `CreatePostDto` 필드 정의
- `UpdatePostDto` 검증 규칙 정의
- `Post` Entity 또는 Interface 설계
- 응답 포맷 표준화
- 에러 응답 표준화
- CORS 설정
- 프론트엔드 API base URL 환경변수화

### 4.7 데이터베이스 및 파일 저장소 도입

현재 저장 방식의 한계:

- 프론트엔드 `localStorage`는 사용자 브라우저별로 분리됨
- 백엔드 메모리 배열은 서버 재시작 시 데이터가 사라짐
- 이미지가 base64로 저장되면 브라우저 저장 용량과 성능 문제가 생김

개선 방향:

- 초기 개발 편의성 기준: SQLite
- 운영 확장 기준: PostgreSQL 또는 MySQL
- 이미지 저장 방식: 서버 로컬 업로드 폴더 또는 오브젝트 스토리지
- 파일 메타데이터는 DB에 저장
- 실제 파일은 별도 저장소에 저장

### 4.8 관리자 영역 부재

현재 `TopHeader.vue`에는 `/admin` 조건부 메뉴가 있으나 실제 관리자 라우트와 화면은 없다.

필요한 기능:

- 관리자 대시보드
- 전체 게시물 통합 목록
- 타입/카테고리/상태/담당자 필터
- 게시/비게시/보관 상태 변경
- 일괄 삭제 또는 일괄 상태 변경
- 관리자 로그인 또는 최소 접근 제어

우선순위:

- 백엔드 연동 전에는 로컬 관리자 화면으로 시작 가능
- 백엔드 연동 후 실제 권한 기능을 붙이는 순서가 적절함

### 4.9 검색과 필터 기능 부족

아카이브 성격의 프로젝트이므로 검색성이 중요하다.

필요한 기능:

- 전체 검색
- 카테고리 필터
- 태그 필터
- 담당자 필터
- 기간 필터
- 디바이스 필터
- 최신순/제목순 정렬
- 검색 조건을 URL query string에 유지

완료 기준:

- 새로고침 후에도 검색 조건 유지
- 검색 결과가 없을 때 명확한 빈 상태 표시
- 컴포넌트와 프로젝트 모두 동일한 검색 경험 제공

### 4.10 테스트 및 품질 관리

현재 기존 문서상 한 차례 빌드/테스트 성공 기록은 있으나, 현재 소스 상태에서는 깨진 파일 때문에 재검증이 필요하다.

필요한 검증:

- `frontend`: `npm run build`
- `backend`: `npm test -- --runInBand`
- 라우터 전체 이동 수동 점검
- 게시물 등록/수정/삭제 수동 점검
- 새로고침 후 데이터 유지 점검

## 7. 리스크 및 주의사항

### 7.1 인코딩 리스크

현재 프로젝트에서 가장 큰 리스크는 한글 인코딩 깨짐이다. 화면 문구만 깨진 것이 아니라 JavaScript 구문과 식별자까지 영향을 받은 파일이 있으므로, 기능 추가보다 복구와 빌드 안정화가 먼저다.

### 7.2 로컬 저장소 의존 리스크

`localStorage`는 개발 초기에 빠르게 기능을 확인하기에는 좋지만, 사용자별 데이터가 분리되고 운영 데이터 공유가 불가능하다. 따라서 백엔드와 DB 연동은 반드시 필요하다.

### 7.3 중복 코드 리스크

컴포넌트 카테고리 페이지가 개별 구현으로 늘어나면 특정 카테고리만 동작이 다르거나 버그가 남기 쉽다. 공통 목록 컴포넌트로 구조를 줄이는 것이 좋다.

### 7.4 경로 오타 리스크

라우터 path는 저장 데이터의 `category`와 직접 연결된다. 오타를 나중에 수정하면 기존 데이터와 URL 호환 문제가 생길 수 있으므로, 초기에 기준표를 확정하는 것이 중요하다.
