# Vue MDN 향후 개발 단계 기획

작성일: 2026-06-01

## 1. 개발 목표

Vue MDN의 향후 개발 목표는 현재의 정적 가이드/더미 데이터 중심 구조를 실제 등록, 조회, 수정, 삭제가 가능한 내부 아카이브 시스템으로 발전시키는 것입니다.

최종 목표:

- 컴포넌트와 프로젝트 산출물을 카테고리별로 등록/관리
- 이미지, 링크, 담당자, 코드, 태그 등 실무 메타데이터 관리
- 관리자 화면에서 콘텐츠 CRUD 수행
- 백엔드 API와 데이터베이스를 통한 영속 저장
- 디자인 가이드와 실제 산출물의 검색/탐색 편의성 강화

## 2. 1단계: 현재 구조 안정화

상태: 실행 완료

우선 화면 이동과 데이터 구조의 불일치를 정리합니다.

### 작업 항목

- 라우터 경로와 상단/좌측 메뉴 경로 통일
- 잘못된 게시글 등록 링크 수정
- `DefaultPage`로 임시 연결된 메뉴의 구현 범위 확정
- 컴포넌트/프로젝트 카테고리 키 네이밍 규칙 확정
- 파일 내 한글 텍스트 및 주석 인코딩 상태 점검
- 공통 버튼, 제목, 리스트, 카드 스타일 정리

### 완료 기준

- 모든 메뉴 클릭 시 의도한 페이지로 이동
- 모든 등록 버튼이 정상 등록 화면으로 이동
- 프론트엔드 빌드 성공
- 주요 화면에서 깨진 문구가 보이지 않음

### 실행 결과

- 상단 헤더 프로젝트 경로를 `/project/megateacher`로 통일
- 좌측 메뉴 스타일가이드 경로를 `/style-guide/...`로 통일
- 좌측 메뉴 프로젝트 기본 경로를 `/project/megateacher`로 통일
- 온라인서점 메뉴 경로를 실제 구현 페이지인 `/project/onlinebookstore`로 통일
- 상품단 라우터 `/component/productlist` 추가
- 이벤트 등록 링크 오타 수정
- 컴포넌트 등록 링크를 절대 경로 `/posting/post?...` 기준으로 정리
- 상품단 페이지의 `postManager` import 누락 수정
- `npm run build` 통과 확인

## 3. 2단계: 게시글 데이터 모델 표준화

상태: 실행 완료

컴포넌트 게시글과 프로젝트 게시글의 공통 필드와 전용 필드를 정리합니다.

### 공통 게시글 필드 후보

- `id`
- `type`: `component` 또는 `project`
- `category`
- `title`
- `description`
- `tags`
- `thumbnailUrl`
- `createdAt`
- `updatedAt`
- `manager`
- `status`: draft, published, archived

### 컴포넌트 게시글 전용 필드 후보

- `deviceTypes`: PC, MO
- `cssCode`
- `htmlCode`
- `jsCode`
- `componentLinks`
- `usageNote`

### 프로젝트 게시글 전용 필드 후보

- `projectBoardUrl`
- `pcUrl`
- `mobileUrl`
- `period`
- `relatedService`
- `projectType`

### 작업 항목

- `PostEditor`와 `ProjectPostEditor`의 필드명 통일
- `ProjectCard`가 표준 데이터 모델을 사용하도록 수정
- `PostPreview`, `ProjectPostPreView`의 props 구조 통일
- `postManager`를 표준 모델 기준으로 정리
- 임시 더미 데이터도 표준 모델로 변환

### 완료 기준

- 등록한 데이터가 목록 카드와 미리보기에서 동일하게 표시
- 컴포넌트/프로젝트 게시글이 같은 저장소에서 타입별로 구분
- 필드명 불일치로 인한 표시 누락 제거

### 실행 결과

- `postManager`에 `normalizePost`, `getAllPosts`, `savePost` 추가
- 게시글 공통 필드를 `type`, `category`, `tags`, `links`, `thumbnailUrl`, `createdAt`, `updatedAt` 중심으로 정리
- 기존 `labels`, `MegaProjectLink`, `PcLink`, `MoLink` 데이터도 읽을 수 있도록 호환 처리
- `PostEditor` 저장 구조를 `type: component`, `tags`, `links` 기반으로 변경
- `ProjectPostEditor` 저장 구조를 `type: project`, `tags`, `links` 기반으로 변경
- `ProjectCard`, `PostPreView`, `ProjectPostPreView`, `PageComponentListItem`이 표준 필드와 기존 필드를 모두 표시하도록 정리
- `npm run build` 통과 확인

## 4. 3단계: 프론트엔드 CRUD 완성

API 연동 전까지는 `localStorage` 기반으로 전체 사용자 흐름을 먼저 완성합니다.

### 작업 항목

- 카테고리별 목록 조회
- 게시글 상세 페이지
- 게시글 수정 진입
- 게시글 삭제
- 삭제 전 확인 팝업
- 등록/수정 후 이동 경로 정리
- 빈 목록 상태 UI
- 이미지 미등록 시 기본 썸네일 처리
- 페이지네이션 또는 더보기 방식 통일

### 완료 기준

- 등록, 목록, 상세, 수정, 삭제가 브라우저 내에서 완결
- 새로고침 후에도 `localStorage` 데이터 유지
- 카테고리별 데이터가 정확히 분리되어 표시

## 5. 4단계: 백엔드 API 고도화

현재 메모리 배열 기반 API를 실제 서비스 API 형태로 정리합니다.

### 작업 항목

- DTO 타입 정의
- 게시글 엔티티 설계
- `PATCH /posts/:id` 또는 `PUT /posts/:id` 수정 API 추가
- `GET /posts/:id` 상세 조회 API 추가
- `GET /posts?type=&category=` 카테고리/타입 조회 API 추가
- 요청 검증 파이프 적용
- 에러 응답 구조 표준화
- CORS 설정
- 프론트엔드 API base URL 환경변수화

### 완료 기준

- 백엔드 단위 테스트 통과
- 프론트엔드에서 API 호출 가능
- Postman 또는 브라우저에서 CRUD API 동작 확인

## 6. 5단계: 데이터베이스 및 파일 저장 연동

운영 가능한 저장 구조를 도입합니다.

### 작업 항목

- 데이터베이스 선택
- 게시글 테이블/컬렉션 설계
- 이미지 업로드 방식 결정
- 썸네일 저장 경로 또는 외부 스토리지 정책 수립
- 마이그레이션 또는 초기 시드 데이터 구성
- 백업/복구 기준 정리

### 데이터베이스 선택 후보

- 빠른 내부 도구 목적: SQLite
- 운영 확장 고려: PostgreSQL 또는 MySQL
- 단순 문서형 메타데이터 중심: MongoDB

### 완료 기준

- 서버 재시작 후에도 게시글 유지
- 이미지 업로드와 조회가 정상 동작
- 개발/운영 환경별 저장 설정 분리

## 7. 6단계: 관리자 기능 구축

관리자 메뉴를 실제 관리 콘솔로 확장합니다.

### 작업 항목

- `/admin` 라우트 추가
- 관리자 대시보드
- 게시글 통합 목록
- 타입/카테고리/상태 필터
- 검색
- 게시/비공개/보관 상태 변경
- 일괄 삭제 또는 일괄 상태 변경
- 관리자 로그인 또는 최소 접근 제어

### 완료 기준

- 관리자 화면에서 모든 게시글 관리 가능
- 일반 사용자 화면과 관리 화면의 역할 분리
- 비공개/보관 게시글이 일반 목록에 노출되지 않음

## 8. 7단계: 검색 및 탐색 편의성 강화

아카이브 성격에 맞게 원하는 산출물을 빠르게 찾을 수 있도록 합니다.

### 작업 항목

- 전체 검색
- 카테고리 필터
- 태그 필터
- 담당자 필터
- 기간 필터
- 디바이스 필터
- 정렬: 최신순, 제목순, 카테고리순
- URL 쿼리스트링으로 필터 상태 유지

### 완료 기준

- 사용자가 프로젝트명, 태그, 담당자로 산출물을 찾을 수 있음
- 새로고침 후에도 검색/필터 상태 유지
- 검색 결과가 없을 때 안내 UI 제공

## 9. 8단계: 디자인 가이드 콘텐츠 보강

현재 비어 있거나 기본 페이지로 연결된 스타일 가이드 항목을 실제 콘텐츠로 채웁니다.

### 작업 항목

- 로고 가이드
- 컬러 시스템
- 폰트 시스템
- 레이아웃 규칙
- 아이콘 규칙
- 공통 컴포넌트 사용 규칙
- 예시 코드와 사용 금지 사례 정리

### 완료 기준

- 스타일 가이드 메뉴가 모두 실제 콘텐츠를 가짐
- 개발자/디자이너가 같은 기준으로 산출물을 제작할 수 있음
- 신규 컴포넌트 등록 시 참고 가능한 규칙이 제공됨

## 10. 9단계: 품질 관리 체계 구축

개발 안정성을 높이기 위한 테스트와 코드 품질 기준을 정리합니다.

### 작업 항목

- 프론트엔드 ESLint/Prettier 설정
- Vue 컴포넌트 테스트 도입 검토
- 핵심 CRUD 유닛 테스트
- 백엔드 서비스/컨트롤러 테스트 확대
- 빌드 검증 스크립트 정리
- Git 작업 전 체크 명령 문서화

### 완료 기준

- `frontend` 빌드가 항상 통과
- `backend` 테스트가 항상 통과
- 주요 CRUD 변경 시 테스트로 회귀 확인 가능

## 11. 우선순위 로드맵

### 단기 우선순위

1. 라우터/메뉴 경로 정리
2. 등록 링크 오류 수정
3. 게시글 데이터 모델 표준화
4. 목록 화면과 `localStorage` 데이터 연결
5. 상세/수정/삭제 플로우 완성

### 중기 우선순위

1. 백엔드 CRUD API 완성
2. 프론트엔드 API 연동
3. 데이터베이스 도입
4. 이미지 업로드 처리
5. 관리자 화면 구축

### 장기 우선순위

1. 검색/필터 고도화
2. 디자인 가이드 콘텐츠 완성
3. 접근 권한 관리
4. 운영 배포 환경 구성
5. 문서화와 유지보수 체계 확립

## 12. 다음 작업 제안

가장 먼저 진행할 작업은 `라우터/메뉴/등록 링크 정리`입니다.

이 작업은 데이터베이스나 백엔드 연동 전에 반드시 안정화해야 하는 기반 작업입니다. 경로가 정리되면 이후 게시글 모델 표준화와 목록 연동 작업을 훨씬 안정적으로 진행할 수 있습니다.
